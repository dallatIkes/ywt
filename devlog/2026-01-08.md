## What I learned

- The good workflow with tokens : when logged in, the token is generated by the server, sent to the client side and stored by the browser. After that every request checks the header for a token, decodes it and checks if it valid. If it's the case, return the requested data, if not, don't.
- My CRUD operations were global, it's good for testing the db but should be replaced by user-scoped endpoints rather than global 
- There are two types of JS in sveltekit : the one that communicates with the browser (directly in the `+page.svelte` file in a `<script></script>` element) and the one that communcates with the server (`+page.server.js`). Generally in the first one we put everything that has to do w/ the DOM and in the second, secrets. It usess 3 canals : `load()` to get data, `actions()` to write and update, and `fetch()` to talk to other services

```python
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/token")

def get_current_user(
    token: str = Depends(oauth2_scheme),
    session: Session = Depends(get_db),
):
    user = decode_access_token(token, session)
    if not user:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Invalid credentials"
        )
    return user
```
